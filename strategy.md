## 概要

カップル向けの記録・管理アプリ。2人で共有しながら、日常の思い出や情報を蓄積していく。

**ソロでも使える**設計が基準：サインアップ時点では自分1人の「スペース（箱）」ができ、招待リンクを送って相手が参加すると共有状態（カップル）になる。つまりカップルは「ソロ＋相手が合流した状態」にすぎない。

## ✅ 決定事項サマリー（現時点）

ここまでで確定した方針の一覧（詳細は各セクション参照）。

**プロダクト設計**

- 名称は **tete（仮）**／ログインは **Googleのみ**（Better Auth）
- データの単位は **「スペース（箱）」**。**ソロが基準**で、招待リンクで相手が合流して2人共有になる
- 招待は **リンク方式**（UUIDv4・有効24h・ワンタイム）
- **複数人モード ＝ 1ユーザーが複数スペースを持てること**（無料=1／単体プラン=複数）
- **公開範囲は機能ごとに固定**（型A 共有／型B 個人／型D 共有＋個人メモ）。ユーザーが切り替えるトグルは作らない
- 相手がアプリ未使用なら `spaces.display_name` のラベルで表現（**案A**。仮ユーザーは作らない）
- 削除は基本 **論理削除**（`deleted_at` / `left_at`）

**技術**

- 構成：**1つの Cloudflare Worker に統合**（同一オリジン。Vue静的アセット＋`/api` Hono）
- スタック：Vue 3 ／ Hono ／ D1 ／ R2 ／ Better Auth
- ORM：**Drizzle**（Prismaは不採用）
- 型共有：`packages/shared` ＋ **Zod** ＋ **Hono RPC**

**残る未確定（要決定）**

- 課金まわり：価格の実数／複数人モードの表ポジション／課金基盤（ストア vs Stripe）／暗号化（→「マネタイズ設計」末尾の論点）

## 機能一覧

機能は公開範囲で3タイプに分かれる（詳細は「技術設計」を参照）。

### 共有（2人で見る）

- **デート記録・予定カレンダー**
  どこに行って何をしたか（過去）と、これからの予定（未来）を1つのカレンダーに表示。
  予定ごとに各自が「個人メモ」を残せる（自分だけに見える一言。Discordのプロフィールメモ的な発想）

- **記念日カレンダー**
  付き合った日・誕生日・各種記念日を月表示で一覧。付き合った日からの日数も表示

- **バケツリスト**
  行きたい場所・やりたいこと（日付なし）を雑多に追加。日付を決めるとカレンダーへ移行できる

### 個人（自分だけ）

- **スキンケアログ × 肌コンディション**
  使用アイテムと肌の調子（5段階）を記録し、ルーティンとコンディションの相関を可視化

- **服装記録**
  前回のデートでどんな服を着たかを記録（かぶり防止）

- **会う前メモ**
  デート前に確認したいことをメモしておける場所

- **プレゼント記録**
  あげた・もらった履歴、次のネタ帳（サプライズ保護のため個人）。あまり増えないのでリスト表示

- **地雷・好み図鑑**
  相手の好きな食べ物・嫌いな食べ物・アレルギー・絶叫系の可否・怒らせてしまうパターンなどを、自分でメモする

### SNS・シェア

- **カード生成**
  デート記録などをいい感じのカードに変換してSNSでシェア。
  スキンケアログなど個人データも、本人が選べばカード化して外部に出せる（相手の見え方は変わらない）

- **写真埋め込み**
  カードや記録に写真を添付可能（R2に保存）

## 技術スタック

**1つの Cloudflare Worker に統合**する（同一オリジン）。Vue をビルドした静的アセットを Worker の `assets` で配信し、`/api/*` を Hono が受ける。Pages を別立てしない。

| 役割           | 技術                                          |
| -------------- | --------------------------------------------- |
| フロントエンド | Vue 3（静的アセットを Worker が配信）          |
| バックエンド   | Hono on Cloudflare Workers（`/api/*`）        |
| データベース   | D1（SQLite）                                  |
| 画像ストレージ | R2                                            |
| 認証           | Better Auth                                   |

### なぜ統合（1 Worker）か＝最優先「安く・早く・ハマらず」

- **コストはほぼ¥0**：Workers（10万req/日）・Workers静的アセット・D1（5GB／読取500万行/日）・R2（10GB＋**下り無料**）はいずれも無料枠で収まる。実費はドメイン代程度。分離（Pages＋Workers）でも金額は変わらないので、差が出るのは手間とリスク。
- **デプロイ1個**：Pages と Workers の2系統を管理しなくてよい。設定・環境変数も1か所。
- **同一オリジン＝CORS不要**：設定ミスでハマる時間がゼロ。
- **Better Auth が素直に動く**：httpOnly Cookie が同一オリジンならそのまま効く（別オリジンだと SameSite／ドメイン設定で確実に一度ハマる）。

## カップル招待設計

### 方式：招待リンク方式

```
自分がリンク生成（例: app.com/invite/xxxxxxxx）
→ 相手にリンクを送る（LINE等）
→ 相手がリンクを踏んでGoogleログイン
→ そのスペース（箱）に参加してペアリング完了
```

- トークンはUUIDv4、有効期限24時間、ワンタイム使用
- LINEで送るだけで完結するため最もスムーズなUX
- 招待は「**どのスペースへの招待か**」を持つ。相手が踏むと `space_members` に相手の行が追加され、同じ `space_id` を共有する＝2人とも同じ箱を覗ける（`spaces` / `space_members` の定義は「技術設計」を参照）

### DB設計

```sql
CREATE TABLE space_invitations (
  id         TEXT PRIMARY KEY,  -- UUIDv4（招待リンクのトークン）
  space_id   TEXT NOT NULL,     -- どのスペースへの招待か
  inviter_id TEXT NOT NULL,     -- 招待した人のuser_id
  expires_at TEXT NOT NULL,     -- 有効期限（24時間）
  used_at    TEXT,              -- 使用済みならタイムスタンプ
  created_at TEXT NOT NULL
);
```

### ペアリング解除

片方が解除したら即時切れる（相手の `space_members` 行を退出扱いにする）。

- 退出は物理削除ではなく `space_members.left_at` を立てる（**論理削除**）。再ペアのクールダウン判定（ペア回し対策）に使う
- スペース自体の削除（別れ→アーカイブ）も `spaces.deleted_at` による論理削除を基本とし、「別れてもデータを切り替える」思想と整合させる

## 🗄️ 技術設計（データモデル）

### 中核概念：user / space / space_members

データの中心に **「スペース（space）」＝1つの関係の全データが入る箱** を置く。
2人の共有ノート1冊のイメージ（LINEのトークルーム／Slackのワークスペースと同じ発想）。

| 用語            | 意味                                              |
| --------------- | ------------------------------------------------- |
| `user`          | アプリにログインする1人の人間（Googleアカウント） |
| `space`         | 1つの関係＝データの箱                             |
| `space_members` | 「どの user がどの space に入ってるか」の対応表   |

```
user ──< space_members >── space ──< 各機能データ
```

- **共有**の正体：太郎と花子が**それぞれ別のuser_idのまま**、両方が同じ `space_id` を指す行を `space_members` に持つ。だから2人とも同じ箱の中身が見える。
- ❌ やってはいけない設計：相手に同じ user_id を共有させる／`users` に `space_id` を1個だけ持たせる（複数スペースで即破綻）。

```sql
-- スペース（1つの関係＝データの箱）
CREATE TABLE spaces (
  id           TEXT PRIMARY KEY,                -- UUID
  started_on   TEXT,                            -- 付き合った日（日数表示の基準）
  kind         TEXT NOT NULL DEFAULT 'couple',  -- 'couple'（2人で使う前提） | 'solo'（1人で使う＝ソロ／複数人モードの箱）
  display_name TEXT,                            -- 相手の表示名（任意）。メンバー1人のソロ時に使用
  created_at   TEXT NOT NULL,
  deleted_at   TEXT                             -- 論理削除（別れ→アーカイブ）
);

-- 所属（誰がどの箱にいるか）= 多対多の中間テーブル
CREATE TABLE space_members (
  user_id   TEXT NOT NULL,
  space_id  TEXT NOT NULL,
  joined_at TEXT NOT NULL,
  left_at   TEXT,                               -- 退出（ペア解除のクールダウン判定に使う）
  PRIMARY KEY (user_id, space_id)               -- ★複合キー＝自動でインデックス＆二重所属防止
);
CREATE INDEX idx_space_members_space ON space_members (space_id);
```

---

### 複数人モデル（スペース中心）

> 「複数人モード」の正体 = **1人のユーザーがスペースを2つ以上持てること**。

- **無料**：スペースは1つだけ
- **単体プラン（複数人モード）**：スペースを複数持てる＋ステルス機能解放
- スペースごとに中身が完全分離されるので、「Aさんの箱」「Bさんの箱」が混ざらない
- **別れ→切り替え**＝アクティブスペースを切り替えるだけ。マネタイズ設計の「別れてもアプリを消さず別の人に切り替える＝チャーンを解約抑止に変える」がそのままデータモデルで表現される

|                | カップルスペース            | 複数人モードのスペース                 |
| -------------- | --------------------------- | -------------------------------------- |
| 実メンバー数   | 2                           | 1（課金者だけ）                        |
| 「相手」の正体 | もう1人のメンバー（実user） | スペース上のプロフィール（名前ラベル） |
| 共有データ     | 双方が見える                | 1人のものなので実質本人だけ            |

**秘匿は“トグル”ではなく“構造”で守る**：カップル相手は、あなたの他のスペースの存在を `space_members` に行が無いため**構造的に観測できない**。アプリ自体の秘匿はステルス機能（アイコン偽装・パスコードロック・通知の相手名非表示）が担う。

**課金との接続（エンタイトルメント）**：

- サブスクは**払った本人（user）に紐づく**
- カップルプランの「1契約で2人解放」は、_「自分が属するスペースの相手がカップル契約中か」_ を実行時に判定して導出する
- ペア回し対策＝再ペアに `space_members.left_at` を使ったクールダウンを課す

**相手がアプリ未使用の場合の持ち方（決定：案A）**：`spaces.display_name` に相手の表示名（文字列ラベル）を持つだけ。仮ユーザー（案B）は作らない。

- 相手が実際に参加するときは招待リンク経由で**実ユーザー**としてメンバーに合流するため、仮ユーザーの事前作成＆身元の紐付け処理が不要
- `display_name` が効くのは**メンバー1人のソロ時のみ**。2人いるカップルでは各自が「相手メンバーのプロフィール名」を見る（見る人によって相手が違うので、スペース固定のラベルは使わない）

| 状態             | 相手の名前の出どころ                     |
| ---------------- | ---------------------------------------- |
| ソロ（相手なし） | 表示しない（`display_name` 空でOK）      |
| 相手アプリ未使用 | `display_name` のラベル                  |
| カップル         | 相手メンバーのプロフィール名             |

### 公開範囲の設計：3つの型

公開範囲は **機能（テーブル）の性質** であって、レコードごと・ユーザーごとの選択にしない。
これにより「公開/非公開の切り替え」がそもそも存在せず、プライバシー漏洩バグが構造的に起きない。

| 型                    | 持つ列                              | 見える範囲              |
| --------------------- | ----------------------------------- | ----------------------- |
| **A. 共有固定**       | `space_id` のみ                     | 同じ箱の全メンバー      |
| **B. 個人固定**       | `space_id` + `owner_user_id`        | 本人だけ                |
| **D. 共有＋個人メモ** | 共有の親 + 個人の子（ユーザーごと） | 親=全員 / メモ=本人だけ |

> ※「C. 個人デフォ＋任意で共有」は検討の結果**不採用**。切り替えが「隠した事実」を相手に観測させ、疑いの種を生むため。

**怪しまれ防止の鉄則：**

1. 「隠す」が観測可能な行為になった瞬間、隠すこと自体が情報になる → だから一律ルール（全員同じ公開範囲）にして個人の「隠す」を消す
2. 個人データは相手画面に**存在ごと出さない**（「🔒ロック項目が3件」のような件数表示も禁止）
3. 課金は公開範囲では差別化しない（「金で隠せる」が疑いの種になる）。差別化は複数人モード＋ステルスで行う
4. （将来トグルを足すなら）「個人デフォ→任意で“開く”」方向だけ。「共有→“隠す”」は観測される減算なので作らない

### 機能別データモデル

| 機能             | 型   | 主なテーブル                                           |
| ---------------- | ---- | ------------------------------------------------------ |
| デート記録・予定 | D    | `date_events`（共有）＋ `date_event_notes`（個人メモ） |
| 記念日           | A    | `anniversaries`                                        |
| バケツリスト     | A    | `bucket_items`                                         |
| スキンケアログ   | B    | `skincare_logs`                                        |
| 服装記録         | B    | `outfit_logs`                                          |
| 会う前メモ       | B    | `pre_date_notes`                                       |
| プレゼント記録   | B    | `gift_logs`                                            |
| 地雷・好み図鑑   | B    | `partner_notes`                                        |
| SNSカード        | 機能 | （保存するなら `cards`。写真はR2参照）                 |

**デート記録・予定（型D＝共有＋個人メモ）**

```sql
-- ① 共有される予定/記録（両人がカレンダーで見る）
CREATE TABLE date_events (
  id               TEXT PRIMARY KEY,
  space_id         TEXT NOT NULL,
  title            TEXT NOT NULL,
  event_date       TEXT NOT NULL,                  -- 予定日 or 行った日
  location         TEXT,
  status           TEXT NOT NULL DEFAULT 'planned',-- 'planned'(予定) | 'done'(実施済み)
  source_bucket_id TEXT,                           -- バケツリスト由来なら紐付け（任意）
  created_by       TEXT NOT NULL,
  created_at       TEXT NOT NULL
);
CREATE INDEX idx_date_events_space ON date_events (space_id);

-- ② 各自の個人メモ（本人だけ見える）
CREATE TABLE date_event_notes (
  event_id   TEXT NOT NULL,
  user_id    TEXT NOT NULL,
  note       TEXT,
  updated_at TEXT NOT NULL,
  PRIMARY KEY (event_id, user_id)                  -- 1予定 × 1ユーザー = メモ最大1個
);
```

- メモ取得は必ず `WHERE event_id=? AND user_id='自分'`。相手のメモは取りに行かない＆「メモあり」マークも出さない。
- 予定削除でメモも消す（`date_event_notes.event_id` に `ON DELETE CASCADE`）。

**記念日（型A・繰り返しは展開しない）**

```sql
CREATE TABLE anniversaries (
  id         TEXT PRIMARY KEY,
  space_id   TEXT NOT NULL,
  title      TEXT NOT NULL,
  event_date TEXT NOT NULL,                        -- 基準日
  recurrence TEXT NOT NULL DEFAULT 'yearly',       -- 'none' | 'monthly' | 'yearly'
  created_at TEXT NOT NULL
);
CREATE INDEX idx_anniversaries_space ON anniversaries (space_id);
```

- 「付き合った日からの日数」は `spaces.started_on` から計算するだけ（テーブル不要）。
- 毎年/毎月の記念日は**全部の発生日を保存しない**。基準日＋繰り返しルールだけ持ち、表示する月の分だけアプリ側で計算する。

**バケツリスト（型A）→ カレンダー連携**

```sql
CREATE TABLE bucket_items (
  id         TEXT PRIMARY KEY,
  space_id   TEXT NOT NULL,
  title      TEXT NOT NULL,
  status     TEXT NOT NULL DEFAULT 'open',         -- 'open' | 'scheduled' | 'done'
  created_at TEXT NOT NULL
);
CREATE INDEX idx_bucket_items_space ON bucket_items (space_id);
```

- 日付を決めて「カレンダーに入れる」と `date_events` を新規作成し、元の項目は `status='scheduled'` に更新（消さずに残す→達成リストにもなる）。

**個人データの共通形（型B）**

スキンケア／服装／会う前メモ／プレゼント／地雷・好み図鑑はすべて同じ形。`owner_user_id` を持つ＝本人だけ。

```sql
-- 例：スキンケアログ
CREATE TABLE skincare_logs (
  id            TEXT PRIMARY KEY,
  space_id      TEXT NOT NULL,
  owner_user_id TEXT NOT NULL,                     -- ★これがある = 個人
  condition     INTEGER,                           -- 肌コンディション 1〜5
  items         TEXT,                              -- 使用アイテム（JSON等）
  logged_on     TEXT NOT NULL,
  created_at    TEXT NOT NULL
);
CREATE INDEX idx_skincare_space_owner ON skincare_logs (space_id, owner_user_id);
```

### テーブル設計の方針（鉄則）

1. **多対多は必ず中間テーブル**（`space_members` / `date_event_notes`）。「片方に列を1個」で済ませない。
2. **IDは全テーブルUUIDで統一**（URLに出ても総当たりされない）。迷えば `crypto.randomUUID()`（v4）、作成順ソートを多用するなら v7。中間テーブルだけは独自IDを持たず複合キー。
3. **共有/個人の境界を列で表現**：共有=`space_id` のみ、個人=`space_id`+`owner_user_id`。最初に分類してから作る。
4. **検索に使う列（WHERE句に来る列）にインデックス**。`space_id` で絞るテーブルには `space_id` インデックス。数十万行でもインデックスがあれば一瞬、無いと全走査で遅い。
5. **繰り返すものは展開して保存しない**：基準日＋ルールを持ち、表示時に計算（記念日・サブスク更新日など）。
6. **削除時の挙動（カスケード）を決める**：スペース削除→中身も消す等。宙に浮くゴミを残さない。
7. **削除は基本“論理削除”**（`deleted_at` / `left_at`）。別れ→アーカイブ→復縁・切り替えに備える。

> 技術スタック・ORM・型共有・案Aなどの確定事項は冒頭「[決定事項サマリー](#決定事項サマリー現時点)」に集約。

## 💰 マネタイズ設計

### 基本方針

**DL無料・基本利用無料**を入口にし、二段階の有料プランを乗せる。
集客と課金のエンジンを分けて考える：

- **集客エンジン**：健全な「tete（2人の絆）」ブランドで TikTok 発信し、Z世代カップルを広く獲得
- **課金エンジン**：複数人モード＋プライバシー機能。課金意欲が高く、チャーンしない層がここで回収を支える

> なぜ重要か：カップルアプリは「別れたら解約」という構造的チャーンを抱える。
> 複数人モードは「別れてもアプリを消さず別の人に切り替える」体験を生み、
> このチャーンを逆に解約抑止に変える。市場も「2人」から「恋愛してる人全員（シングル含む）」に広がる。

---

### プラン構成

```
無料：DL／基本利用（1組 or 1人、写真◯枚、カード基本テンプレ）
  ↓
単体プラン：複数人を分けて記録 ＋ ステルス（アイコン偽装／パスコードロック／通知の相手名非表示）
  ↓
カップルプラン：1契約で2人とも解放（容量UP・プレミアムカード・基本機能の上限解放を含む）
```

| プラン   | 価格（仮） | 1人あたり | 主な価値                     | 想定ターゲット             |
| -------- | ---------- | --------- | ---------------------------- | -------------------------- |
| 無料     | ¥0         | —         | 基本記録・基本カード         | 全員（入口）               |
| 単体     | ¥480/月    | ¥480      | 複数人モード＋ステルス       | ソロ・複数同時進行・プロ層 |
| カップル | ¥680/月    | ¥340      | 1契約で2人解放・全機能上限UP | 付き合っているカップル     |

※年額は実質値引き。買い切り（2年分一括）は **ローンチ記念・期間限定のみ** とし、常設しない
（常設するとLTVが頭打ちになり継続課金＝本線を自分で潰すため）。

### カップルプラン（1契約で2人解放）の狙い

単なる割引ではなく、5つの効果が同時に効くのが肝：

1. **ARPU向上**：2人に別々に単体を売るより、カップル1契約の方が1ユーザーあたり単価が高い
2. **無料側をタダで獲得＆布教**：払わない側の獲得コストがゼロ、かつ友達に広める＝CAC実質半減
3. **課金＝愛情表現に変換**：「彼女に贈るギフト」フレームで、払う側の「ケチに見られたくない」心理を逆手に取る
4. **相互ロックイン**：2人で同じデータに投資するため片方だけ辞めづらい
5. **チャーン耐性**：2人巻き込むぶん離脱が重くなる

> マーケは「彼氏が彼女に贈るギフト」として振るが、**プロダクトは性別を固定しない**（中立に設計）。

### 🧮 価格の絶対鉄則

> **単体価格 ＜ カップル価格 ＜ 単体価格 × 2**

- この範囲を守れば、ソロユーザーが2アカウント作ってカップルプランを買っても
  支払額は単体より高くなる＝**漏れではなくボーナス**。本人は「2人ぶん使えてお得」と得した気持ちになり、契約数が増える。
- 逆に **カップルを単体より安くすると自爆**：全員が2アカ作ってカップルへ流れ、
  「裏ワザ」が拡散した瞬間にそれが正規価格化してARPUが崩壊する。

### 🚨 本当に塞ぐべき穴：スロット転売／ペア回し

solo2アカは無害。実害が出るのは **1契約の無料スロットを他人に売る・貸す（Netflix型共有）**。

対策（設計で潰す）：

- プレミアムは **払った本人のアカウントに紐づく**。相手側は **ペアが有効な間だけ** 解放
- **再ペアにクールダウン**（解除後◯日は再ペア不可 or 月◯回まで）でペア回し営業を物理的に封じる
- 無料側の解放範囲を「共有に必要な範囲」に絞り、容量・ステルス等の個人価値機能は本人のみとする手もある

## 💗 ネーミング検討まとめ

### 決定（仮）：**tete（テテ）**

「手と手」をつなぐ＝2人の絆を表すネーミング。アイコンの「手つなぎ」モチーフと
セットで意味が完成する、名前×ビジュアル一体型のブランディング。

### 命名の戦略整理

アプリ名には大きく2つの戦略があり、本プロジェクトは **A：バズ前提型** を採用。

| 戦略                      | 例                      | 特徴                                                                                          | 向いているケース                             |
| ------------------------- | ----------------------- | --------------------------------------------------------------------------------------------- | -------------------------------------------- |
| **A：バズ前提型**         | tete / setlog           | 名前は世界観・ロゴで勝負。中身はSNSで説明。短く洗練・かぶりにくいが、バズらないと気づかれない | **自分でTikTok等で発信していく場合**         |
| **B：検索・ストア自走型** | ふたりログ / ふたりごと | 名前で中身が伝わる。バズらなくても検索流入を拾える。オシャレ度はやや下がる                    | 広告予算が限られ、自然流入で積み上げたい場合 |

→ 本プロジェクトは **自分でTikTok発信していく前提** のため、A戦略の「tete」を採用。

### tete のメリット

- 4文字で覚えやすく、打ちやすい（SNSで布教しやすい）
- `#tete` のハッシュタグがオシャレにキマる
- 韓国っぽい響きで Z世代・女子（カップルアプリの主役層）に刺さる
- 「テテって何？」という引っかかり＝興味のフックになる
- 既存イメージに縛られず世界観を自由に作れる

### 注意点 / 対策

- **検索で埋もれるリスク**：BTSのテテ（V）と被る懸念があったが、調査の結果ほぼ競合なし。
  ただし発信時は「カップルアプリのtete（テテ）」とカタカナ読みもセットで打ち出す。
- **商標・名称かぶり**：アルファベット3文字は要チェック（リリース前に確認）。
