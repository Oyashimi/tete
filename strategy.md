---

## 🎀 ネーミング・ブランディング担当ペルソナ

> **女性カフェ店員（流行に敏感）**
> ふわふわしたものや可愛いものが好き。Z世代のトレンドやSNS（TikTok）の感覚を持ち、

## 概要

カップル向けの記録・管理アプリ。2人で共有しながら、日常の思い出や情報を蓄積していく。

---

## 機能一覧

### 記録系

- **スキンケアログ × 肌コンディション**
  使用アイテムと肌の調子（5段階）を記録し、ルーティンとコンディションの相関を可視化

- **デートプラン記録**
  どこに行って何をしたかを記録

- **服装記録**
  前回のデートでどんな服を着たかを記録（かぶり防止）

- **会う前メモ**
  デート前に確認したいことをメモしておける場所

- **プレゼント記録**
  あげた・もらった履歴、次のネタ帳として活用（サプライズ保護のため非共有）

### 共有系

- **記念日・付き合った日カウンター**
  付き合い始めた日からの日数を表示

- **バケツリスト**
  行きたい場所・やりたいことを2人で共有し、達成したらチェック

- **地雷メモ**
  怒らせてしまうパターンや注意点を記録

- **好み図鑑**
  好きな食べ物・嫌いな食べ物・アレルギー・絶叫系の可否などを記録

### SNS・シェア系

- **カード生成**
  デート記録などをいい感じのカードに変換してSNSでシェア

- **写真埋め込み**
  カードや記録に写真を添付可能

## 認証設計

### ログイン方法

**Googleログインのみ**を採用。

- パスワードを自前DBで持たないため流出リスクがゼロ
- Googleが2FA・ブルートフォース対策を担う
- 実装がシンプル

### セッション管理

Better Authに委任。以下をデフォルトで担保：

- セッショントークンのローテーション
- httpOnly Cookie（JSからトークンを盗めない）
- CSRF対策

## カップル招待設計

### 方式：招待リンク方式

```
自分がリンク生成（例: app.com/invite/xxxxxxxx）
→ 相手にリンクを送る（LINE等）
→ 相手がリンクを踏んでGoogleログイン
→ ペアリング完了
```

- トークンはUUIDv4、有効期限24時間、ワンタイム使用
- LINEで送るだけで完結するため最もスムーズなUX

### DB設計

```sql
CREATE TABLE couple_invitations (
  id         TEXT PRIMARY KEY,  -- UUIDv4
  inviter_id TEXT NOT NULL,     -- 招待した人のuser_id
  expires_at TEXT NOT NULL,     -- 有効期限（24時間）
  used_at    TEXT,              -- 使用済みならタイムスタンプ
  created_at TEXT NOT NULL
);
```

### ペアリング解除

片方が解除したら即時切れる。

## 💰 マネタイズ設計

### 基本方針

**DL無料・基本利用無料**を入口にし、二段階の有料プランを乗せる。
集客と課金のエンジンを分けて考える：

- **集客エンジン**：健全な「tete（2人の絆）」ブランドで TikTok 発信し、Z世代カップルを広く獲得
- **課金エンジン**：複数人モード＋プライバシー機能。課金意欲が高く、チャーンしない層がここで回収を支える

> なぜ重要か：カップルアプリは「別れたら解約」という構造的チャーンを抱える。
> 複数人モードは「別れてもアプリを消さず別の人に切り替える」体験を生み、
> このチャーンを逆に解約抑止に変える。市場も「2人」から「恋愛してる人全員（シングル含む）」に広がる。

---

### プラン構成

```
無料：DL／基本利用（1組 or 1人、写真◯枚、カード基本テンプレ）
  ↓
単体プラン：複数人を分けて記録 ＋ ステルス（アイコン偽装／パスコードロック／通知の相手名非表示）
  ↓
カップルプラン：1契約で2人とも解放（容量UP・プレミアムカード・基本機能の上限解放を含む）
```

| プラン   | 価格（仮） | 1人あたり | 主な価値                     | 想定ターゲット             |
| -------- | ---------- | --------- | ---------------------------- | -------------------------- |
| 無料     | ¥0         | —         | 基本記録・基本カード         | 全員（入口）               |
| 単体     | ¥480/月    | ¥480      | 複数人モード＋ステルス       | ソロ・複数同時進行・プロ層 |
| カップル | ¥680/月    | ¥340      | 1契約で2人解放・全機能上限UP | 付き合っているカップル     |

※年額は実質値引き。買い切り（2年分一括）は **ローンチ記念・期間限定のみ** とし、常設しない
（常設するとLTVが頭打ちになり継続課金＝本線を自分で潰すため）。

---

### カップルプラン（1契約で2人解放）の狙い

単なる割引ではなく、5つの効果が同時に効くのが肝：

1. **ARPU向上**：2人に別々に単体を売るより、カップル1契約の方が1ユーザーあたり単価が高い
2. **無料側をタダで獲得＆布教**：払わない側の獲得コストがゼロ、かつ友達に広める＝CAC実質半減
3. **課金＝愛情表現に変換**：「彼女に贈るギフト」フレームで、払う側の「ケチに見られたくない」心理を逆手に取る
4. **相互ロックイン**：2人で同じデータに投資するため片方だけ辞めづらい
5. **チャーン耐性**：2人巻き込むぶん離脱が重くなる

> マーケは「彼氏が彼女に贈るギフト」として振るが、**プロダクトは性別を固定しない**（中立に設計）。

---

### 🧮 価格の絶対鉄則

> **単体価格 ＜ カップル価格 ＜ 単体価格 × 2**

- この範囲を守れば、ソロユーザーが2アカウント作ってカップルプランを買っても
  支払額は単体より高くなる＝**漏れではなくボーナス**。本人は「2人ぶん使えてお得」と得した気持ちになり、契約数が増える。
- 逆に **カップルを単体より安くすると自爆**：全員が2アカ作ってカップルへ流れ、
  「裏ワザ」が拡散した瞬間にそれが正規価格化してARPUが崩壊する。

---

### 🚨 本当に塞ぐべき穴：スロット転売／ペア回し

solo2アカは無害。実害が出るのは **1契約の無料スロットを他人に売る・貸す（Netflix型共有）**。

対策（設計で潰す）：

- プレミアムは **払った本人のアカウントに紐づく**。相手側は **ペアが有効な間だけ** 解放
- **再ペアにクールダウン**（解除後◯日は再ペア不可 or 月◯回まで）でペア回し営業を物理的に封じる
- 無料側の解放範囲を「共有に必要な範囲」に絞り、容量・ステルス等の個人価値機能は本人のみとする手もある

---

### 課金まわりの未確定論点（要決定）

- [ ] 単体／カップルの **実数**（¥480・¥680は仮。もっと攻めるか）
- [ ] 複数人モードの **表ポジション**：「複数同時進行（健全寄り・TAM最大）」で行くか、ステルスを全面に出すか
      （浮気・キャバ嬢を名指しするとブランド汚染＆App Store審査リスク。マーケでは名指ししない方針）
- [ ] 課金基盤：ストア課金（手数料15〜30%）前提で価格を逆算。Webサブスク（Stripe）併用はストア規約に注意
- [ ] データ漏洩対策：複数人・秘密データは流出が即致命傷。強い暗号化・ロック・ステルスを「機能」かつ「保険」として実装

## 💗 ネーミング検討まとめ

### 決定（仮）：**tete（テテ）**

「手と手」をつなぐ＝2人の絆を表すネーミング。アイコンの「手つなぎ」モチーフと
セットで意味が完成する、名前×ビジュアル一体型のブランディング。

### 命名の戦略整理

アプリ名には大きく2つの戦略があり、本プロジェクトは **A：バズ前提型** を採用。

| 戦略                      | 例                      | 特徴                                                                                          | 向いているケース                             |
| ------------------------- | ----------------------- | --------------------------------------------------------------------------------------------- | -------------------------------------------- |
| **A：バズ前提型**         | tete / setlog           | 名前は世界観・ロゴで勝負。中身はSNSで説明。短く洗練・かぶりにくいが、バズらないと気づかれない | **自分でTikTok等で発信していく場合**         |
| **B：検索・ストア自走型** | ふたりログ / ふたりごと | 名前で中身が伝わる。バズらなくても検索流入を拾える。オシャレ度はやや下がる                    | 広告予算が限られ、自然流入で積み上げたい場合 |

→ 本プロジェクトは **自分でTikTok発信していく前提** のため、A戦略の「tete」を採用。

### tete のメリット

- 3文字で覚えやすく、打ちやすい（SNSで布教しやすい）
- `#tete` のハッシュタグがオシャレにキマる
- 韓国っぽい響きで Z世代・女子（カップルアプリの主役層）に刺さる
- 「テテって何？」という引っかかり＝興味のフックになる
- 既存イメージに縛られず世界観を自由に作れる

### 注意点 / 対策

- **検索で埋もれるリスク**：BTSのテテ（V）と被る懸念があったが、調査の結果ほぼ競合なし。
  ただし発信時は「カップルアプリのtete（テテ）」とカタカナ読みもセットで打ち出す。
- **商標・名称かぶり**：アルファベット3文字は要チェック（リリース前に確認）。

### アプリアイコン

- **モチーフ**：手つなぎ（小指つなぎ／指切り）＋ ハート ＋ きらめき
- **カラー**：くすみピンク（ミルクティーベージュ系も候補）＝トレンドど真ん中
- 名前「tete」＝手と手、をアイコンのビジュアルで補完する設計
